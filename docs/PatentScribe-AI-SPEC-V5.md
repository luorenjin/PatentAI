## 2. Problem Statement（问题陈述）

**我们在解决什么问题？**

工程师撰写专利交底书时，核心痛点不是"不会写"，而是：

1. **不知道代理人需要什么信息**：初稿常缺失技术方案对比、新颖性论证、实施细节，导致代理人打回
2. **来回沟通成本高**：平均每份交底书经历 3-5 轮邮件往返，耗时 2-4 周
3. **传统工具是"上传-下载"模式**：工程师上传初稿 → 工具输出优化版 → 工程师下载 → 发现还有问题 → 再上传... 效率低下
4. **输出格式不标准**：不同工程师写的交底书格式各异，代理人需要额外时间理解结构

**谁遇到这个问题？** 企业中高频产出专利的研发工程师（芯片/AI/软件领域，每周 1 份以上）

**频率如何？** 专利产出高峰期，工程师每月产出 2-4 份交底书

**不解决的代价？**
- 企业专利布局节奏滞后
- 代理人资源被低质量初稿占用（代理人 30% 时间花在"打回-指导补充"上）
- 工程师对专利撰写产生抵触情绪

**Evidence（证据）:**
- User research: [待补充，建议至少 5 次工程师访谈 + 3 次代理人访谈]
- Behavioral data: [待补充，交底书打回率、平均修改轮次、工程师耗时]
- Support signal: [待补充，IP 部门反馈、代理人投诉]
- Competitive signal: 智慧芽 AI 助手、PatentPal 等仍是"上传-下载"工具，非 AI Native 体验，且输出格式不标准
- **Qwen 3.6-plus 验证**：已完成召回测试，10 份真实交底书分析结果可行（代理人评估：一次通过率提升显著）

## 5. User Personas & Stories（用户画像与核心用户故事）

### Primary Persona

**李明**，芯片设计工程师，5 年经验，某半导体企业，每月需提交 2-3 份专利交底书，对专利语言不熟悉，依赖代理人反馈迭代。

### Secondary Persona

**王律师**，专利代理人，8 年经验，每天审阅 5-8 份交底书，最大痛点是"工程师写的初稿缺关键信息，来回沟通浪费时间"。

### 核心用户故事

**Story 1**: 作为工程师，我想通过对话式 AI 界面粘贴文本或上传 .docx/.pdf，以便 AI 立即分析并告诉我哪里需要补充。

**Acceptance Criteria（验收标准）**:
- [ ] Given 工程师粘贴文本或上传 .docx/.pdf，when 点击"开始分析"，then 30 秒内 AI 完成深度分析
- [ ] 分析完成后，AI 以对话形式输出"诊断报告"（不是静态文档）：列出缺失信息和建议补充的内容
- [ ] 右侧实时预览区显示优化后的交底书（高亮标注 AI 补充/修改的内容）
- [ ] Given 文件格式不支持，when 上传，then 显示友好错误提示并说明支持格式
- [ ] Given .pdf 文件，when 上传，then 系统按配置处理（默认：直接提交大模型；开启"本地处理"：使用 Rust 库预处理）

**Story 2**: 作为工程师，我想在对话界面中回答 AI 的追问，以便动态补充交底书缺失的关键信息。

**Acceptance Criteria（验收标准）**:
- [ ] AI 根据分析结果动态生成追问（数量不固定，按需生成）
- [ ] 追问以对话气泡形式出现，工程师在输入框中回答
- [ ] 每次回答后，AI 实时更新右侧预览区的交底书内容（无需手动刷新）
- [ ] 工程师可以随时选择"这部分我已经写清楚了，不需要补充"跳过某条追问
- [ ] AI 判断信息已充足时，主动提示"当前交底书信息已完整，可以下载终稿"（**仅作建议，工程师可继续补充**）

**Story 3**: 作为工程师，我想随时终止补充并下载当前版本的交底书，以便在代理人催交时不等 AI 判断"信息已充足"。

**Acceptance Criteria（验收标准）**:
- [ ] 对话界面始终显示"下载当前版本"按钮
- [ ] 下载格式为 .docx，保留原稿格式 + AI 优化内容标注（高亮/批注）
- [ ] 下载前弹出选项："是否包含 AI 追问回答记录"（默认不含）
- [ ] Performance: 生成 .docx 文件 ≤ 10 秒

**Story 4**: 作为工程师，我想看到 AI 补充/修改的内容被高亮标注，以便快速核验技术细节。

**Acceptance Criteria（验收标准）**:
- [ ] AI 补充的内容用蓝色高亮标注
- [ ] AI 修改的原文用黄色高亮标注
- [ ] 每处高亮旁附批注说明理由（如"基于工程师回答补全"或"原文描述不清晰，已重写"）
- [ ] 提供"移除所有高亮"按钮（下载前可一键清除标注）

**Story 5**: 作为代理人，我想看到 AI 优化后的交底书中清楚标注哪些是 AI 补充的内容，以便我快速定位需要核验的部分。

**Acceptance Criteria（验收标准）**:
- [ ] 下载的 .docx 文件中，AI 补充内容用蓝色高亮 + 批注
- [ ] 批注中包含补充理由和置信度（如"基于工程师回答补全，置信度：高"）
- [ ] 提供"代理人视图"开关：开启后仅显示高亮内容，方便快速核验

**Story 6**: 作为工程师，我想在设置中配置 PDF 处理方式（默认直接提交大模型，可开启本地 Rust 库处理），以便根据我的需求选择处理方式。

**Acceptance Criteria（验收标准）**:
- [ ] 设置页面包含"PDF 处理方式"选项
- [ ] 选项 1：**直接提交大模型处理**（默认，无需预处理）
- [ ] 选项 2：**开启本地处理**（使用 Rust 库 pdf-extract 预处理 .pdf）
- [ ] 配置立即生效，无需重启应用
- [ ] 设置自动保存（下次启动自动加载）

**Story 7**: 作为工程师，我希望 AI 输出的交底书严格遵循标准格式（8 个固定章节 + 2 个专项输出），以便代理人快速理解技术方案。

**Acceptance Criteria（验收标准）**:
- [ ] AI 输出严格遵循 8 个固定章节（发明名称、技术领域、背景技术及现有缺陷、发明目的、技术方案、有益效果、附图说明、具体实施方式）
- [ ] AI 输出包含 2 个专项输出（专利挖掘与布局建议、需工程师补齐的关键改进建议）
- [ ] 第五部分"技术方案"使用工程师口吻（非法律术语）
- [ ] 第六部分"有益效果"使用表格格式（维度 | 技术突破点 | 效果与价值体现）
- [ ] "专利挖掘与布局建议"使用表格格式（布局方向 | 保护策略 | 建议权利要求架构）
- [ ] "需工程师补齐的关键改进建议"使用问答式（Q1/Q2...）

## 6. Solution Overview（方案概述）

### 产品定位

**AI Native 专利交底书搭档** —— 基于 **Tauri 框架**，不是传统的"上传-下载"工具，而是**对话式 AI 界面 + 实时预览**的沉浸式体验。

工程师与 AI 对话，AI 动态引导补充信息，右侧实时预览交底书优化效果。最终输出一份**标准交底书**（严格遵循 8 个固定章节 + 2 个专项输出），代理人一次通过率提升 2 倍以上。

### 核心流程

```
工程师粘贴文本 / 上传 .docx/.pdf
              ↓
   [配置] PDF 处理方式
   - 默认：直接提交大模型
   - 开启本地处理：Rust 库预处理
              ↓
    AI 深度分析（创新性 / 新颖性 / 实用性）
    【强制遵循输出格式：8 章节 + 2 专项】
              ↓
  AI 以对话形式输出"诊断报告"
  （列出缺失信息和建议补充的内容）
              ↓
    工程师在对话界面回答 AI 追问
              ↓
  AI 实时更新右侧预览区的交底书
  （高亮标注补充/修改的内容）
  【自动遵循输出格式规范】
              ↓
   循环：AI 判断是否需要继续追问
   （按需动态生成，非固定轮次）
              ↓
  AI 判断"信息已充足"（仅作建议）
  OR 工程师选择"提前终止"
              ↓
      下载最终版 .docx（含/不含 AI 标注）
```

### 动态追问机制（非固定轮次）

| 场景 | AI 行为 | 工程师行为 | 终止条件 |
|------|---------|-----------|---------|
| 初稿分析完成 | AI 输出诊断报告 + 生成追问（数量按需） | 回答 / 跳过 | — |
| 工程师回答后 | AI 更新交底书 + 判断是否需要继续追问 | 继续对话 | — |
| AI 判断信息已充足 | AI 主动提示"当前交底书信息已完整，可以下载终稿"（**仅作建议**） | 选择"继续补充"或"下载终稿" | 工程师确认下载 |
| 工程师想提前终止 | — | 点击"提前终止"按钮 | 工程师主动终止 |

**关键设计原则**：
- 追问数量**不固定**，由 AI 根据交底书缺失程度动态生成
- 工程师**随时可以终止**（不需要等到 AI 判断"信息已充足"）
- 每次回答后，AI **实时更新**右侧预览区的交底书（工程师立即看到优化效果）
- AI 判断"信息已充足"**仅作建议**，工程师可继续补充或手动终止

### PDF 处理配置（可选配置，默认直接提交大模型）

| 配置项 | 选项 A：直接提交大模型（默认） | 选项 B：开启本地处理 |
|--------|---------------------------|-------------------|
| 处理方式 | 将 .pdf 作为二进制数据提交给千问 API（支持多模态） | 使用 Rust 库（pdf-extract）提取文本 + 布局信息，再提交大模型 |
| 适用场景 | 简单排版 .pdf（文字可选中） | 复杂排版 .pdf（表格、公式、多栏） |
| 优点 | ✅ 无需预处理，速度快<br>✅ 利用大模型多模态能力（可识别图片、公式） | ✅ 解析质量高（Rust 库性能强）<br>✅ 不依赖大模型多模态能力 |
| 缺点 | ❌ 复杂排版（表格、公式）可能解析错误 | ❌ 需要额外开发（Rust 库集成）<br>❌ 图片、公式识别能力弱于大模型 |
| 配置位置 | 设置页面 → "PDF 处理方式" → **默认选中** | 设置页面 → "PDF 处理方式" → 可选 |

**默认策略**：
- **默认**：直接提交大模型处理（无需预处理，速度快）
- **可选**：工程师可在设置页面开启"本地处理"（使用 Rust 库 pdf-extract 预处理）
- 配置立即生效，无需重启应用

### Key Design Decisions（关键设计决策）

| Decision | 选择 | 理由 | 取舍 |
|----------|------|------|------|
| 桌面框架 | **Tauri**（Rust + Web 前端） | 包体积小（~5MB vs Electron ~150MB）、性能高、Rust 生态支持 PDF 处理 | 前端技术栈需适配 Tauri API；Rust 学习曲线高 |
| 交互模式 | 对话式 AI 界面（左侧对话、右侧预览） | AI Native 体验，工程师与 AI "协作"完成优化 | 开发复杂度高于传统"上传-下载"工具 |
| 追问机制 | 动态按需生成（非固定轮次） | 简单交底书可能 1 轮搞定，复杂的交底书可能需要多轮，固定轮次不灵活 | AI 判断"信息已充足"可能不准确 → 仅作建议 + 工程师可手动终止 |
| 实时预览 | 每次回答后立即更新右侧预览区 | 工程师需要**即时反馈**（看到 AI 的补充效果），而不是"回答一堆问题后再看结果" | 对前端性能要求高（大型 .docx 渲染可能卡顿） |
| 大模型选择 | 千问（Qwen3.6-plus / qwen-max / qwen-plus） | 中文专利语料表现强，API 成本可控，已验证可行 | 放弃 GPT-4 的多语言优势 |
| 文件格式支持 | .docx / .pdf / 粘贴文本 | 灵活，工程师可以选择最方便的方式 | .pdf 解析质量依赖配置（默认直接大模型 vs 本地 Rust 库） |
| PDF 处理方式 | **可选配置**（默认：直接提交大模型；可选：开启本地处理） | 灵活适配不同场景（简单 .pdf → 默认；复杂排版 .pdf → 开启本地处理） | 需要维护两套处理逻辑（但用户可自选，降低复杂度） |
| 输出文件格式 | .docx | 代理人工作流依赖 Word | 暂不支持 .pdf 输出（V2） |
| AI 标注方式 | 高亮 + 批注 | 代理人需快速定位 AI 补充内容 | 增加文档复杂度 |
| PDF 处理默认策略 | **直接提交大模型**（无需预处理） | 大多数 .pdf 文字可选中，直接提交大模型速度快；复杂排版 .pdf 可手动开启本地处理 | 复杂排版 .pdf 默认处理可能不准确 → 用户可手动切换 |
| **输出格式规范** | **强制 8 章节 + 2 专项输出** | 标准化输出，代理人快速理解；降低工程师与代理人沟通成本 | 对 AI 输出约束强，可能增加 AI 生成难度 |

### Prompt 工程关键设计（V5 更新）

**分析 Prompt（含输出格式强制约束）**：
```
你是一位资深专利工程师与专利代理人协作专家。请根据我提供的原始技术构思/草稿，协助优化并深度挖掘专利交底书中的新颖性、创造性与实用性，输出一份可直接交付给专利代理人的《标准技术交底书》。

【输出格式要求】（必须严格按此顺序与标题输出，不得增删或合并章节）
一、发明名称
二、技术领域
三、背景技术及现有缺陷（需明确对比现有技术的结构/工艺/性能痛点，量化或定性描述缺陷，为后续创造性做铺垫）
四、发明目的（直击背景缺陷，明确本方案要解决的技术问题）
五、技术方案（核心保护点）
六、有益效果（新颖性、创造性、实用性体现）
七、附图说明（供绘图参考）
八、具体实施方式（提供至少2个实施例，包含具体参数范围、装配关系、工作流程或控制逻辑）

【专项输出要求】（置于交底书正文末尾）
🔍 专利挖掘与布局建议（供专利代理人参考）
🛠️ 需工程师补齐的关键改进建议（问答式）

【撰写原则与强制约束】
1. 语体与结构：
   - 第五部分"技术方案"禁用过度法言法语。请以"工程师口吻"按"结构关系 → 工程意图 → 保护重心"分层提炼，重点突出拓扑差异、控制逻辑与给代理人的撰写提示，降低沟通成本。
   - 其他部分严格使用专利规范术语（如"进一步地""优选地"），禁用广告化、模糊化表述。
2. 逻辑链条：突出"结构创新→功能实现→效果提升"的因果关系。创造性挖掘必须聚焦：①现有技术无法兼顾的技术矛盾 ②本方案如何通过特定结构/参数/工艺打破该矛盾 ③产生的非预期协同效果。
3. 信息处理：若输入信息缺失，严格标注"【待工程师补充】"，严禁虚构核心技术细节、公差或实验数据。
4. 格式强制：
   - 第六部分"有益效果"必须采用表格，列名为：维度 | 技术突破点 | 效果与价值体现。
   - "🔍 专利挖掘与布局建议"必须采用表格，列名固定为：布局方向 | 保护策略 | 建议权利要求架构。下方可附简短审查应对提示。
   - "🛠️ 需工程师补齐的关键改进建议"必须采用问答式（Q1/Q2...），每个问答需明确包含：①缺失项对创造性/可实施性的影响 ②建议补充的具体形式 ③对应的权利要求布局价值。

现在，请分析以下专利交底书初稿：

[交底书初稿内容]
```

### Output Format Validation（输出格式校验）

为确保 AI 输出严格遵循 8 章节 + 2 专项格式，Tauri 后端需实现**输出格式校验逻辑**：

| 校验项 | 校验规则 | 失败处理 |
|--------|---------|---------|
| 章节完整性 | 输出必须包含 8 个固定章节（一、发明名称 ... 八、具体实施方式） | 提示 AI"输出格式不完整，缺少【章节名】"，重新生成 |
| 专项输出完整性 | 输出必须包含 2 个专项输出（🔍 专利挖掘与布局建议、🛠️ 需工程师补齐的关键改进建议） | 提示 AI"输出格式不完整，缺少【专项输出名】"，重新生成 |
| 第六部分表格格式 | "有益效果"必须使用表格（维度 \| 技术突破点 \| 效果与价值体现） | 提示 AI"第六部分格式错误，必须使用表格"，重新生成 |
| 专利挖掘建议表格格式 | "🔍 专利挖掘与布局建议"必须使用表格（布局方向 \| 保护策略 \| 建议权利要求架构） | 提示 AI"专利挖掘建议格式错误，必须使用表格"，重新生成 |
| 工程师补齐建议问答格式 | "🛠️ 需工程师补齐的关键改进建议"必须使用问答式（Q1/Q2...） | 提示 AI"工程师补齐建议格式错误，必须使用问答式"，重新生成 |
| 缺失信息标注 | 若信息缺失，必须标注"【待工程师补充】"，严禁虚构 | 提示 AI"发现虚构内容，请标注【待工程师补充】"，重新生成 |

**校验流程**：
```
AI 生成交底书
      ↓
Tauri 后端校验输出格式
      ↓
校验通过 → 更新右侧预览区
校验失败 → 提示 AI"格式错误，请重新生成" → 重新校验（最多 3 次）
```

### B. 交底书质量评分维度详细定义

| 维度 | 5 分 | 3 分 | 1 分 |
|------|------|------|------|
| 创新性 | 技术方案有明确创新点，与现有方案差异显著 | 有创新点但描述模糊 | 无明确创新点或与现有方案无差异 |
| 新颖性 | 现有技术对比充分，差异点量化描述 | 有对比但不够充分 | 缺少现有技术对比 |
| 实用性 | 技术方案可实施、有验证数据 | 理论可行但缺少验证 | 实施路径不清晰 |
| **输出格式规范性** | **严格遵循 8 章节 + 2 专项输出，格式完全符合规范** | **格式基本符合，但存在少量偏差** | **格式严重不符合规范** |

### C. UI/UX 设计要点（AI Native 体验）

**对话式界面设计原则**：
- AI 消息使用**对话气泡**（不是静态文本）
- 工程师回答使用**输入框 + 发送按钮**（类似 ChatGPT）
- 右侧预览区**实时更新**（每次回答后 500ms 延迟渲染，避免卡顿）
- AI 主动提示使用**系统消息气泡**（如"信息已充足，可以下载终稿"【仅作建议】）

**实时预览区设计原则**：
- 使用**虚拟滚动**（大型 .docx 可能超过 50 页）
- 高亮标注使用**颜色编码**（蓝色 = AI 补充，黄色 = AI 修改）
- 提供**缩放控件**（工程师可以放大查看技术细节）
- 提供"代理人视图"开关（开启后仅显示高亮内容）
- **强制显示 8 章节 + 2 专项输出**（即使 AI 输出格式错误，预览区也按标准格式渲染）

**设置页面设计原则**：
- PDF 处理方式：**默认选中"直接提交大模型处理（推荐）"**，可选"开启本地处理（Rust 库）"
- 配置立即生效，无需重启应用
- 提供"推荐"标签（默认模式适合大多数场景；复杂 .pdf 建议开启本地处理）

### D. Tauri 技术栈优势

| 维度 | Tauri（Rust + Web 前端） | Electron（Node.js + Chromium） |
|------|------------------------|-------------------------------|
| 包体积 | ~5MB | ~150MB |
| 内存占用 | 低（Rust 无 GC） | 高（Chromium + Node.js） |
| 性能 | 高（Rust 原生性能） | 中（JavaScript 性能瓶颈） |
| Rust 生态 | 强（pdf-extract、docx-rs 等库） | 弱（需调用外部进程） |
| 学习曲线 | 高（Rust 语言难度） | 低（前端技术栈熟悉） |
| 跨平台 | Windows / macOS / Linux | Windows / macOS / Linux |
| **输出格式校验** | **强（Rust 原生性能，可快速校验大型 .docx）** | **中（JavaScript 性能瓶颈，校验大型 .docx 可能卡顿）** |

**选择 Tauri 的核心原因**：
1. **包体积小**：~5MB vs Electron ~150MB，用户下载安装更友好
2. **性能高**：Rust 原生性能，处理大型 .docx/.pdf 更快
3. **Rust 生态**：pdf-extract、docx-rs 等库支持，PDF/.docx 解析无需调用外部进程
4. **内存占用低**：Rust 无 GC，长期运行更稳定
5. **输出格式校验性能强**：Rust 原生性能可快速校验大型 .docx 的输出格式（AI 生成的交底书可能 50+ 页）

**校验流程**：

```

       /       \
     Yes        No

            重新生成（最多 3 次）
              ↓

         No        Yes
```
